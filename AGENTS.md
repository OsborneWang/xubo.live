# AGENTS.md

本文件用于约束 AI 编码助手在本项目中的行为。进入本项目后，支持读取 `AGENTS.md` 的工具应遵守以下规则。

## Safety Rules

- 在执行任何高风险操作前，必须先向用户说明风险并获得明确确认。
- 高风险操作包括但不限于：
  - 删除文件或目录，尤其是批量删除、递归删除、`rm -rf`。
  - 覆盖大量文件或重写关键配置。
  - 执行 `git reset --hard`、`git clean`、强制 checkout、rebase、force push 等可能丢失改动的 Git 操作。
  - 安装、升级、删除依赖，或运行来自未知来源的脚本。
  - 数据库迁移、清空数据、修改生产数据。
  - 修改 `.env`、密钥、证书、token、生产配置等敏感文件。
- 未经用户明确授权，不允许执行 `rm -rf`。
- 未经用户明确授权，不允许回滚、覆盖或删除用户已有改动。
- 修改文件前，应优先查看相关文件内容，避免盲改。
- 如果发现工作区存在用户未提交的改动，应避免覆盖，并在必要时提醒用户。

## Workflow Rules

- 对于简单问题，可以直接回答或读取必要文件后回答。
- 对于涉及代码修改的任务，应先简要说明将要做什么。
- 对于中等或复杂开发任务，必须先给出 TODO / 实施计划，等待用户确认后再开始修改代码。
- 对于需求不明确的任务，应先提问澄清，而不是自行假设。
- 执行命令前，应说明命令目的；高风险命令必须先获得确认。
- 所有工作和文档都应留痕：需求、计划、关键决策、执行记录、变更总结、验证方式和后续建议，应汇总维护到 `docs/work-log.md`。
- 新增或调整需求、开发计划、架构决策、重要约定时，应优先更新对应文档，并在 `docs/work-log.md` 追加记录。
- 修改完成后，应总结：
  - 修改了哪些文件。
  - 做了什么变更。
  - 如何验证。
  - 是否还有后续建议。
  - 是否已更新工作记录。

## Sub-Agent Work Mode

- 对于中等或复杂任务，AI 助手可以采用 sub-agent 工作模式，将任务拆分为 Planner、Researcher、Implementer、Reviewer、Docs Keeper 等角色。
- 当前环境未必支持真实并行子代理时，可由单个助手模拟多个角色完成分工。
- 可并行的只读检查、文件读取、搜索和验证应尽量并行执行。
- 每个角色的关键结论、实施结果和验证结果应汇总到 `docs/work-log.md`。
- 采用 sub-agent 工作模式时，应在开始前说明角色分工、执行范围和需要用户确认的风险点。

## Coding Rules

- 尽量保持改动最小化，只修改与当前任务相关的文件。
- 遵循项目现有代码风格、目录结构和命名规范。
- 不随意引入新依赖；如确需引入，应先说明原因并征求确认。
- 不随意大规模重构；如确需重构，应先给出计划并征求确认。
- 优先修复根因，而不是只做表面绕过。
- 新增功能时，应尽量补充必要的类型、测试或验证方式。

## Command and Tool Usage

- 可以直接执行低风险的只读命令，例如：`pwd`、`ls`、`find`、`rg`、`git status`、读取文件等。
- 可以执行常规验证命令，例如测试、lint、build，但如果命令会修改文件或环境，应先说明并确认。
- 长时间运行的服务命令，例如 `npm run dev`，应先询问用户是否需要启动，或建议用户在独立终端运行。
- 不要在没有必要时访问网络、安装包或下载脚本。

## Privacy and Secrets

- 不要打印、复制或泄露密钥、token、证书、cookie、私有配置等敏感信息。
- 如果必须检查敏感配置，只说明是否存在、格式是否合理，不要在回复中展示完整内容。
- 不要将敏感信息写入日志、文档、测试快照或示例代码。

## Git Rules

- 未经用户要求，不要自动提交、push、创建 tag 或改写 Git 历史。
- 修改前后可使用 `git status` 查看状态。
- 如果需要提交，应先展示变更摘要，并等待用户确认 commit message。

## Communication

- 默认使用中文回复。
- 回复应简洁、明确，重要路径用代码格式标出。
- 遇到不确定的信息，应说明不确定性并建议验证方式。
